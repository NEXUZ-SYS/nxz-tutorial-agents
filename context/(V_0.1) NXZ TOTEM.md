# Product Briefing: NxzGo — Modo TOTEM (Self-Service)

> Gerado em 2026-03-16 por análise do codebase (v5.13.1).

## 1. Visão Geral

O **modo Totem** é a interface de autoatendimento do NxzGo, projetada para que o próprio cliente realize seu pedido sem intervenção de um atendente. É utilizado em totens de autoatendimento (tablets ou dispositivos dedicados) posicionados no salão ou na entrada do estabelecimento.

O Totem compartilha a mesma base de código do PDV, mas com comportamento adaptado para uso pelo consumidor final: interface simplificada, sem acesso a funções administrativas, com proteção de tela (screensaver) durante inatividade e navegação por drawer bloqueada. Internamente é identificado por `configs.type !== 'caixa'` (tipicamente `'totem'`), resultando em `configs.isPdv === false`.

O Totem é ideal para restaurantes de serviço rápido, fast-foods, praças de alimentação e qualquer operação que busque reduzir filas e otimizar o atendimento com autoatendimento digital.

## 2. Público-Alvo

- **Consumidores finais** — Clientes que fazem pedidos diretamente no totem, sem interação com atendentes
- **Gerentes de operação** — Que configuram o totem via painel Odoo (whitelabel, produtos, pagamentos)

### Necessidades do consumidor:
- Interface intuitiva e visual para montar pedidos sem treinamento
- Processo de pagamento simplificado (apenas métodos eletrônicos)
- Feedback claro sobre status do pedido
- Opção de pagar no caixa quando há dificuldade com pagamento eletrônico

## 3. Funcionalidades Core

### 3.1 Tela de Atração (Screen Saver)

Quando o totem fica ocioso, exibe um slideshow de imagens promocionais configuradas via whitelabel (`whiteLabel.savers`). Cada imagem tem duração configurável. O texto "Toque para iniciar" é sobreposto. Qualquer toque na tela retorna à tela de boas-vindas.

O screensaver é ativado após **20 segundos** de inatividade na tela inicial, gerenciado pelo `TimerManager`.

### 3.2 Tela de Boas-Vindas (StartCustomerOrder)

- Logo do estabelecimento (centralizada)
- Mensagem de boas-vindas customizável (`whiteLabel.welcome_message` ou "Bem vindo à")
- Tempo estimado de espera (`whiteLabel.time_value` ou "03 min") — ocultável via `whiteLabel.hidden_time`
- Botão "Novo Pedido" (texto customizável via `whiteLabel.new_order_button`)
- Se o usuário está em modo `BOTH`: botão para alternar para KDS

### 3.3 Identificação do Cliente

Mesmos três modos do PDV (simples, identificação, unificado), porém:
- **Sem seletor de tabela de preços** (exclusivo do PDV)
- Interface otimizada para toque direto pelo cliente

### 3.4 Catálogo de Produtos

Layout adaptativo baseado no tipo de dispositivo:

- **Tablet**: Sidebar lateral com categorias + grid de produtos à direita. Barra de busca toggle. Footer com botão de cancelar e carrinho animado com badge de quantidade.
- **Phone**: Chips de categorias em scroll horizontal + grid de produtos. Resumo do carrinho no footer.

Funcionalidades:
- Busca com debounce de 300ms
- Leitor de código de barras (se `configs.useBarcode`)
- Multiplicador de quantidade (1x, 2x, 3x, 5x, 10x, 20x)
- Modal de produto com detalhes e personalização

### 3.5 Personalização de Produtos

Tela fullscreen para customização via BOM:
- Seleção de tamanhos, sabores, adicionais
- Controle de quantidade (+/-)
- Preço atualizado em tempo real
- Auto-adição quando há apenas uma opção

### 3.6 Carrinho / Revisão

Steps visuais: **Revisão** → Nota Fiscal → Pagamento → Confirmação

O carrinho do Totem é mais simplificado que o do PDV:
- **Sem edição de preço** por item
- **Sem seletor de tabela de preços**
- **Sem modal de cupom manual** (apenas input de código se `configs.discount.useDiscount`)
- Lista de itens com quantidade, notas e remoção
- Subtotal, desconto, gorjeta e total

### 3.7 Sistema de Pagamento (Diferenças do PDV)

A diferença mais significativa entre Totem e PDV está no pagamento:

| Aspecto | Totem | PDV (Caixa) |
|---|---|---|
| Journals exibidos | Todos juntos, sem separação | Online vs Offline com senha |
| Pagamentos offline | Ocultos ou protegidos | Disponíveis com senha |
| "Pagar no balcão" | Sim (se `enableSendToKDSAsDraft`) | Não |
| Divisão de conta | Sim (se `useSplit`) | Sim |
| Badge de pagamento | Sim (se `whiteLabel.show_payment_type_badge`) | Sim |

A opção **"Pagar no balcão"** é exclusiva do Totem: o pedido é salvo como rascunho no KDS e o cliente é direcionado ao caixa para pagamento presencial. Isso evita abandono de pedido quando o cliente tem dificuldade com o pagamento eletrônico.

### 3.8 Timer de Inatividade

O `TimerManager` monitora inatividade em todas as telas do PDV/Totem:
- Qualquer toque na tela reseta o timer
- Ao expirar, retorna à tela inicial (`StartCustomerOrder`) ou ao screensaver
- Configurável via `configs.disableTimer`

### 3.9 Nota Fiscal

Mesmo fluxo do PDV com possibilidade de:
- Não emitir NF
- Imprimir NF (NFC-e ou SAT)
- Enviar NF por e-mail
- Configurável para pular a seleção (`configs.skipNfSelection`)

### 3.10 Tela de Sucesso

- Mensagem customizável (`whiteLabel.success_message`)
- GIF animado de sucesso
- Countdown de **10 segundos** para retorno automático à tela inicial
- Botão "Imprimir nota" (se impressora configurada)
- Botão "Fazer novo pedido"
- Impressão automática se configurada (`autoPrinterEnabled`)

## 4. Stack Técnica

### Frameworks & Linguagens
- **React Native 0.81.1** + **TypeScript 5.8**
- **React Navigation 7** (Native Stack para fluxo linear)
- **React Native Paper 5** (UI components)
- **React Native Reanimated 4** (animações do screensaver e transições)

### Pagamento
Utiliza os mesmos provedores do PDV, mas com foco em métodos eletrônicos:
- Sumup, Cielo, PagBank, Stone, Pagarme (APIs)
- PIX (via QR code exibido na tela)
- Dinheiro tipicamente oculto (configurável)

### White Label
O sistema de whitelabel é especialmente importante para o Totem:
- **Cores**: `general` (cor primária, header, footer) e `home_screen` (gradientes, botões, textos)
- **Imagens**: Logo, screensavers, GIF de sucesso
- **Textos**: Boas-vindas, botão de pedido, labels de identificação, mensagem de sucesso, label do pager
- **Comportamentos**: Ocultar tempo de espera, ocultar impressão, pular seleção de NF

## 5. Arquitetura

### Contextos

```
AuthProvider
  └── StoreProvider (whitelabel + configs + catálogo)
       └── Pdv (TimerManager + Timer + CartProvider)
            └── CartProvider
                 └── PDV Routes Stack (mesma stack, comportamento adaptado)
```

O Totem usa **exatamente a mesma stack de navegação** do PDV. A diferenciação é feita em runtime via `configs.isPdv` e `configs.type`:

```typescript
// Exemplo de diferenciação no PaymentType/Content.tsx
if (!isPdv) {
  // Totem: mostra todos os journals juntos
} else {
  // Caixa: separa online vs offline
}
```

### Drawer bloqueado

No app routes, quando `configs.type !== 'caixa'`:
```
swipeEnabled: false
drawerType: 'front'
```
Isso impede que o cliente acidentalmente abra o menu administrativo.

## 6. Integrações

Mesmas do PDV, com destaque para:

| Integração | Relevância no Totem |
|---|---|
| **Firebase/Supabase** | Essencial para "Pagar no balcão" (draft → KDS) |
| **Provedores de pagamento** | Foco em eletrônicos (sem dinheiro por padrão) |
| **SAT/NFC-e** | Fiscal com opção de e-mail (cliente digita no totem) |
| **Odoo ERP** | Whitelabel, catálogo, configurações |
| **Sentry** | Monitoramento (totens operam sem supervisão constante) |

## 7. Regras de Negócio

- **Sem acesso a pagamentos offline** por padrão — o Totem é projetado para autoatendimento com pagamento eletrônico
- **Timer de inatividade obrigatório** — Evita que o totem fique travado em uma tela intermediária
- **Screensaver como atrator** — Exibe promoções e convida o próximo cliente
- **"Pagar no balcão"** é a válvula de escape — Quando o pagamento eletrônico falha, o pedido é salvo como draft e o cliente paga no caixa
- **Drawer bloqueado** — O cliente não pode acessar configurações, gestão de pedidos ou outras telas administrativas
- **Sem edição de preço** — O consumidor final não pode alterar preços ou acessar tabelas de preço alternativas
- **Recuperação pós-crash** — Mesmo no totem, pedidos pendentes são recuperados automaticamente
- **Modo imersivo** — A barra de navegação do Android é ocultada para experiência fullscreen

## 8. Deploy & Distribuição

### Dispositivos-alvo
- Tablets Android (tipicamente 10" ou maior) em modo quiosque
- Dispositivos dedicados de autoatendimento

### Build
- `yarn project playstore` para build genérico (distribuição Play Store ou APK)
- Suporta update OTA via cron `Upgrade`

### Configuração
- Toda personalização é feita via **painel Odoo** (whitelabel, produtos, pagamentos)
- Configuração inicial no dispositivo via tela `WelcomeConfiguration` → `SignIn` → `Configuration`

## 9. Estado Atual & Observações

### Pontos fortes
- O sistema de whitelabel permite personalização profunda sem alteração de código
- O screensaver com imagens configuráveis é eficaz como ferramenta de marketing no ponto de venda
- A opção "Pagar no balcão" é uma solução elegante para o problema de abandono no totem
- Timer de inatividade garante que o totem nunca fica "travado"

### Áreas de atenção
- O Totem e o PDV compartilham a mesma codebase com diferenciação condicional — isso significa que qualquer mudança no fluxo PDV pode impactar o Totem
- Verificações de `isPdv` estão espalhadas por múltiplos componentes, aumentando a complexidade de manutenção
- O screensaver depende de URLs de imagens configuradas no whitelabel — se não configurado, a tela volta imediatamente
- A acessibilidade para diferentes perfis de consumidor (idosos, PcD) não é explicitamente tratada no código atual
- Modo imersivo (ocultar barra de navegação Android) é essencial para UX de quiosque
