# Product Briefing: NxzGo — Modo KDS (Kitchen Display System)

> Gerado em 2026-03-16 por análise do codebase (v5.13.1).

## 1. Visão Geral

O **modo KDS (Kitchen Display System)** é o painel de exibição e gerenciamento de pedidos para a cozinha e áreas de produção. Diferente do PDV e Totem que focam na criação de pedidos, o KDS foca na **preparação e expedição**, recebendo pedidos em tempo real e permitindo que a equipe de cozinha acompanhe, organize e avance o status de cada item.

O KDS recebe pedidos de múltiplas fontes simultaneamente: PDV (caixa), Totem (self-service), iFood (delivery) e Open Delivery (Keeta, 99Food, VCQPad). Todos os pedidos convergem para uma única interface onde a equipe de cozinha pode visualizar, filtrar e gerenciar a produção.

É identificado pelo `user.mode === 'KDS'` ou `user.mode === 'BOTH'` (quando o mesmo dispositivo alterna entre PDV e KDS).

## 2. Público-Alvo

- **Cozinheiros e chapeiros** — Visualizam itens a preparar, marcam como prontos
- **Expedidores** — Conferem pedidos completos e despacham para entrega/retirada
- **Gerentes de cozinha** — Monitoram tempo de preparo e gargalos
- **Equipe de delivery** — Gerenciam pedidos iFood/Open Delivery com ações específicas (aceitar, rejeitar, disputar)

### Necessidades principais:
- Visualização clara e em tempo real de todos os pedidos pendentes
- Acompanhamento individual de itens dentro de cada pedido
- Ações rápidas: marcar como pronto, aceitar pedido iFood, imprimir ticket
- Filtragem por categorias relevantes à estação de trabalho
- Histórico de pedidos finalizados

## 3. Funcionalidades Core

### 3.1 Navegação por Abas (Bottom Tabs)

O KDS utiliza navegação por abas na parte inferior:

| Aba | Descrição | Badge |
|---|---|---|
| **Home** | Pedidos ativos/em produção | Contador de pedidos pendentes |
| **Delivery** | Pedidos de delivery (iFood/Open Delivery) | Contador de entregas pendentes |
| **Histórico** | Pedidos finalizados | Contador de finalizados |

A aba **Delivery** só aparece quando `company.ifood_id` está configurado.

### 3.2 Home — Pedidos Ativos

A tela principal exibe cards de pedidos em FlatList, ordenados por:
1. **Pedidos iFood não confirmados** (prioridade máxima — aguardando aceite)
2. **Pedidos por data de criação** (`created_at`)

Cada card de pedido (`Order` component — 1362 linhas) exibe:
- **Header**: Número do pedido, canal de origem (POS, iFood, Open Delivery), horário, tipo (dine-in/takeaway)
- **Itens**: Lista de produtos com quantidades e stage individual
- **Ações**: Botões contextuais baseados no estado do pedido

Filtros:
- Exclui pedidos finalizados e de delivery
- Filtro de pedidos agendados (iFood `SCHEDULED`) com toggle e badge de contagem
- Pedidos "pending finish" (todos os itens prontos, aguardando finalização)

### 3.3 Gestão de Stages (Estágios de Produção)

Cada item de produção (`IProduction`) possui um `stage`:
- `''` (vazio) — Aguardando
- `'progress'` — Em preparação
- `'done'` — Pronto
- `'confirmed'` — Confirmado/expedido

O operador do KDS pode avançar itens individualmente ou todos de uma vez. O cron `ProductionItem` gerencia a transição automática de stages.

### 3.4 Filtro de Categorias

Configurável via `kdsConfig`:
- `use_category_filter` — Ativa filtragem
- `categories` — Lista de categorias relevantes para esta estação
- `category_filter_action` — Comportamento para itens fora da categoria:
  - `'no_warning'` — Exibe normalmente
  - `'warning'` — Exibe com indicador visual de alerta
  - `'hide'` — Oculta itens fora da categoria
  - `'block'` — Bloqueia interação com itens fora da categoria

Isso permite que diferentes estações (grill, bar, sobremesas) vejam apenas os itens relevantes.

### 3.5 Integração iFood

O KDS é o ponto central para gerenciamento de pedidos iFood:

- **Aceitar/Rejeitar** pedidos com modal de confirmação
- **Disputar** pedidos com motivo
- **Status do merchant** — Indicador no header (aberto/fechado/alerta) com modal de detalhes
- **Auto-aceite** — Configurável via `AutoAcceptIfoodOrders`
- **Pedidos agendados** — Exibe separadamente com filtro toggle
- **Cron `IFoodStatus`** — Sincroniza status do merchant periodicamente

### 3.6 Integração Open Delivery

Suporte a pedidos de plataformas Open Delivery:
- **Keeta** (by Google)
- **99Food**
- **VCQPad**
- Outros via padrão Open Delivery

### 3.7 Impressão de Tickets

Cada pedido pode ser impresso em formato:
- **Ticket** — Resumo para a cozinha
- **Sticker** — Etiqueta para identificação
- **DANFE** — Nota fiscal (quando aplicável)

Impressão automática na chegada de novo pedido (se `autoPrint` configurado).

### 3.8 Histórico

A aba de histórico exibe pedidos finalizados ordenados por data (mais recentes primeiro). O filtro de categorias é desabilitado (`'block'`) para que todos os itens sejam visíveis no histórico.

### 3.9 Header do KDS

- Nome do usuário e empresa
- Logo da loja
- **Botão "PDV"** — Troca para modo PDV (se `user.mode === 'BOTH'`), com modal de confirmação
- **Indicador iFood** — Status da loja no iFood (aberto/fechado) com modal de detalhes
- **Filtro de agendados** — Toggle com badge de contagem
- **Relógio** — Atualizado a cada 20 segundos

## 4. Stack Técnica

### Frameworks
- **React Native 0.81.1** + **TypeScript 5.8**
- **React Navigation 7** (Bottom Tabs)
- **React Native Paper 5** (UI)

### Real-time
- **Firebase Realtime Database** — Listener `onValue` para atualizações instantâneas de pedidos
- **Supabase** — Alternativa via Postgres change events (INSERT/UPDATE/DELETE)
- **EventEmitter** (`OrderManager`) — Distribui eventos de atualização para componentes React

### Background Jobs (KDS)
| Cron | Frequência | Função |
|---|---|---|
| `ExcludeOrder` | Configurável | Remove pedidos antigos/finalizados |
| `IFoodStatus` | 30h | Sincroniza status do merchant iFood |
| `ProductionItem` | Configurável | Gerencia transição de stages |

### Estrutura de dados em tempo real

```
Firebase: /erp-<seller>/<company_id>/orders[<branch>]/<orderId>
Supabase: orders (table) WHERE channel = '<path>'
```

Cada pedido contém:
- Metadados (número, data, canal, tipo)
- Array de itens com stage individual
- Dados de delivery (quando aplicável)
- `isDraft` / `isOpenTab` flags
- `nf` (dados fiscais)
- `stage` / `note` — Campos mantidos pelo KDS (preservados durante atualizações do PDV)

## 5. Arquitetura

### Contextos

```
AuthProvider
  └── StoreProvider (configs + kdsConfig)
       └── Order (componente montado em contexts/index.tsx para modo KDS/BOTH)
            └── OrderManager (singleton EventEmitter)
                 └── Kds (wrapper context — atualmente pass-through)
                      └── KDS Routes (Bottom Tabs)
```

### `OrderManager` — O coração do KDS

O `OrderManager` é um singleton que:
1. Subscreve ao Firebase/Supabase via `Database.listener()`
2. Recebe atualizações em tempo real
3. Emite eventos `'update'` via `EventEmitter`
4. Componentes React subscrevem via `OrderManager.on('update', callback)`

### Fluxo de um pedido no KDS

1. PDV/Totem cria pedido → `Sale.send()` salva no Firebase/Supabase
2. Firebase emite `onValue` → `OrderManager` recebe atualização
3. `OrderManager.emit('update')` → Componente `Home` re-renderiza
4. Operador KDS marca itens como prontos → `Database.update()` salva `stage` no Firebase
5. Quando todos os itens estão `done` → Pedido move para "pending finish"
6. Operador finaliza → Pedido move para aba `History`
7. Cron `ExcludeOrder` remove pedidos antigos do histórico

## 6. Integrações

| Integração | Finalidade | Detalhes |
|---|---|---|
| **Firebase Realtime DB** | Fonte principal de pedidos | Listener `onValue`, `onDisconnect` para status |
| **Supabase** | Alternativa ao Firebase | Postgres change events via realtime channel |
| **iFood** | Pedidos de delivery | Aceitar/rejeitar, status merchant, auto-aceite |
| **Open Delivery** | Pedidos de outras plataformas | Keeta, 99Food, VCQPad |
| **Odoo ERP** | Configurações KDS, categorias | Via `Session.getPosConfig` e API |
| **Impressora** | Tickets e etiquetas de produção | V1/V2/V3 conforme dispositivo |
| **Sentry** | Monitoramento de erros | Essencial para operação 24/7 |
| **WebSocket (Odoo Bus)** | Comandos remotos | Restart, clear cache, debug, send CFe |

## 7. Regras de Negócio

- **Pedidos iFood não confirmados têm prioridade visual** — Aparecem primeiro na lista para evitar timeout de aceite
- **Filtro de categorias permite estações especializadas** — Cada KDS pode ver apenas itens de sua estação (grill, bar, sobremesas)
- **Stages são gerenciados individualmente por item** — Um pedido com hambúrguer e suco pode ter o suco pronto antes do hambúrguer
- **Campos `stage` e `note` do KDS são preservados** — Quando o PDV atualiza um pedido no Firebase, os campos de controle do KDS não são sobrescritos
- **Auto-print na chegada** — Novos pedidos podem ser impressos automaticamente para produção
- **Máximo de 100 pedidos no histórico** — O `clearHistory()` mantém apenas os 100 mais recentes para performance
- **Pedidos draft/comanda** são visíveis no KDS — Permitem preparação antecipada mesmo sem pagamento
- **Health check do Firebase** — Write + Read + Delete com timeout de 5s cada, garantindo conectividade
- **`onDisconnect`** — Quando o KDS perde conexão, automaticamente marca o POS como offline no Firebase

## 8. Deploy & Distribuição

### Dispositivos-alvo
- Tablets Android (cozinha) — tipicamente com suportes/braços articulados
- Dispositivos resistentes a ambiente de cozinha (calor, umidade)

### Configuração
- Modo KDS é definido no cadastro do usuário no Odoo (`user.mode = 'KDS'`)
- Configuração de categorias filtradas é feita via Odoo
- Múltiplos KDS podem operar simultaneamente (diferentes estações = diferentes filtros)

### Requisitos de rede
- **Conexão estável com Firebase/Supabase é crítica** — O KDS depende de real-time para funcionar
- WebSocket support verificado via `NativeModules` (`supportsWebSocket`)
- Fallback sem real-time: KDS não funciona adequadamente sem conexão

## 9. Estado Atual & Observações

### Pontos fortes
- Arquitetura real-time robusta com Firebase + fallback Supabase
- Filtro de categorias por estação é uma funcionalidade diferenciadora
- Integração profunda com iFood (aceite, rejeição, disputa, status)
- O componente `Order` (1362 linhas) é extremamente completo, cobrindo todos os canais
- `onDisconnect` do Firebase garante visibilidade de status de conectividade

### Áreas de atenção
- O componente `Order` com 1362 linhas é o maior do projeto e poderia ser modularizado
- O contexto `Kds` é atualmente um pass-through vazio — toda lógica está no `OrderManager` singleton
- Dependência forte de conectividade — Não há modo offline para o KDS
- O cron `IFoodStatus` roda a cada 30 horas, o que pode ser longo demais para detectar mudanças de status
- Performance com grande volume de pedidos simultâneos não é explicitamente otimizada (FlatList sem virtualização adicional)
