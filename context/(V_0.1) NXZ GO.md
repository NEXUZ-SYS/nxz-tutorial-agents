# Product Briefing: NxzGo — Modo PDV (Caixa)

> Gerado em 2026-03-16 por análise do codebase (v5.13.1).

## 1. Visão Geral

O **modo PDV (Ponto de Venda)** — internamente referenciado como `caixa` — é a interface de operação para atendentes e caixas humanos. É o modo mais completo do NxzGo, oferecendo controle total sobre vendas, pagamentos, gestão de pedidos abertos, impressão fiscal e integração com múltiplos provedores de pagamento.

O PDV funciona como um sistema de frente de caixa completo para restaurantes, lanchonetes e estabelecimentos de food service, conectado ao ERP Odoo como backend. O operador tem acesso a funcionalidades avançadas como edição de preço, aplicação de descontos, seleção de tabela de preços, gestão de comandas abertas e divisão de conta — recursos que não estão disponíveis no modo Totem (self-service).

O PDV é identificado pela configuração `configs.type === 'caixa'` e `configs.isPdv === true`, definidos a partir do campo `iface_pos_type` da configuração do POS no Odoo.

## 2. Público-Alvo

- **Atendentes de caixa** em restaurantes, lanchonetes, cafeterias e food courts
- **Gerentes de loja** que precisam de acesso administrativo (descontos, cancelamentos, gestão de pedidos)
- **Operadores de balcão** que recebem pedidos presenciais e processam pagamentos

### Necessidades principais:
- Registro rápido de pedidos com busca e leitor de código de barras
- Flexibilidade para editar preços, aplicar descontos e cupons
- Gestão de comandas abertas (open tabs) e pedidos rascunho
- Múltiplas formas de pagamento incluindo dinheiro, cartão e PIX
- Emissão de nota fiscal (NFC-e / SAT)

## 3. Funcionalidades Core

### 3.1 Fluxo de Pedido

O fluxo completo de um pedido no PDV segue estas etapas:

1. **Tela Inicial** (`StartCustomerOrder`) — O operador inicia um novo pedido. Pode recuperar pedidos pendentes após crash/restart.
2. **Identificação do Cliente** (`CustomerIdentification`) — Captura nome e/ou documento do cliente. Três modos: simples (só nome), identificação (busca por CPF/email/telefone via PartnerMuk), ou unificado (ambos).
3. **Opção de Consumo** (`DineOptions`) — "Para comer aqui" ou "Para levar" (configurável).
4. **Catálogo de Produtos** (`ProductsList`) — Navegação por categorias com layout adaptativo (phone/tablet). Suporte a leitor de código de barras com multiplicador de quantidade.
5. **Personalização** (`Customization`) — Customização de produtos via BOM (Bill of Materials) — tamanhos, sabores, adicionais.
6. **Carrinho** (`ShoppingCart`) — Revisão do pedido com edição de itens, desconto, código de vendedor, seleção de mesa, gorjeta.
7. **Pagamento** — Seleção de método, processamento e confirmação.
8. **Envio e Confirmação** — Submissão ao backend, impressão e tela de sucesso.

### 3.2 Sistema de Pagamento

O PDV separa os métodos de pagamento em duas categorias:

- **Pagamentos Online (Eletrônicos)**: Cartão de crédito, débito, voucher e PIX — processados via integração com maquininhas ou APIs
- **Pagamentos Offline**: Dinheiro e métodos manuais — protegidos por senha de acesso, exclusivos do modo caixa

Funcionalidades exclusivas do PDV:
- **Divisão de conta** (`useSplit`) — Permite pagamento parcial com múltiplos métodos
- **Troco** (`PaymentChange`) — Cálculo automático para pagamento em dinheiro
- **Código de transação** (`PaymentCode`) — Registro de código externo de transação
- **Cancelamento MSiTef** — Disponível no drawer para terminais MSiTef

### 3.3 Gestão de Pedidos

- **Comandas abertas** (`PaidLater` / `SaveDraft`) — Pedidos podem ser salvos como rascunho ou comanda aberta para pagamento posterior
- **Gestão de Pedidos** (`OrderManagement`) — Tela administrativa acessível via drawer: lista pedidos em aberto, permite retomar, cancelar ou imprimir
- **Recuperação de pedidos** — Após crash/restart, o app detecta pedidos em cache e oferece recuperação

### 3.4 Nota Fiscal

- **Seleção de modelo** (`NFStatus`) — NFC-e (modelo 65) ou SAT/CF-e (modelo 59)
- **E-mail de NF** (`NFStatusEmail`) — Envio por e-mail ao cliente
- **Impressão automática** — Configurável para impressão automática via DANFE
- **Integração SAT** — Geração de XML CF-e, envio ao equipamento SAT local, com retries automáticos

### 3.5 Controles Avançados

- **Edição de preço por item** — Habilitado quando `configs.discount.usePrice === true`, com limites configuráveis
- **Seleção de tabela de preços** — Quando há múltiplas pricelists configuradas
- **Desconto/cupom** — Input de código de cupom com validação automática
- **Código do vendedor** — Rastreamento de vendedor por pedido
- **Seleção de mesa** — Integração com planta do salão (floor/table hierarchy do Odoo)
- **Gorjeta** — Percentual ou valor fixo configurável

### 3.6 Navegação

- **Drawer lateral com slide** — Acesso rápido a: PDV, KDS (se modo `BOTH`), Produtos (controle de disponibilidade), Gestão de Pedidos, Cancelamento MSiTef
- Drawer habilitado para swipe, diferente do modo Totem onde é bloqueado

## 4. Stack Técnica

### Linguagens & Frameworks
- **React Native 0.81.1** com **TypeScript 5.8**
- **React 19.1.0** com Context API para state management
- **React Navigation 7** (Native Stack + Drawer + Bottom Tabs)
- **React Native Paper 5** para componentes UI

### Provedores de Pagamento (modo Playstore/genérico)
| Provedor | Códigos | Mecanismo |
|---|---|---|
| Sumup | `SUMUP` | Native SDK Android |
| Sumup PIX | `SUMUP_PIX` | Variante PIX do Sumup |
| Cielo | `CCD/CCC/CCR/CCPX` | API REST com polling |
| PagBank | `PCD/PCC/PCR/PPIX` | Bluetooth Adapter |
| Stone | `SCD/SCC/SCR/SCPX` | API REST com polling (requer senha) |
| Pagarme | — | API REST com polling |
| Dinheiro | `CASH` | Offline, callback imediato |

### Impressão
- **V1**: RawBT / EscPOS (padrão para modo Playstore)
- **V2**: BLE / Network TCP / USB (modo `direct`)
- **V3**: Impressora nativa SmartPOS (Cielo, Stone, PagBank)

### Backend & Dados
- **Odoo ERP** como sistema central (sessões POS, produtos, categorias, vendas)
- **Firebase Realtime Database** para sincronização em tempo real de pedidos
- **Supabase** como alternativa ao Firebase
- **AsyncStorage** para cache local e recuperação de crash

## 5. Arquitetura

### Contextos principais do PDV

```
AuthProvider (autenticação global)
  └── StoreProvider (catálogo, configurações, whitelabel)
       └── Pdv (wrapper com TimerManager + CartProvider)
            └── CartProvider (estado do pedido atual)
                 └── PDV Routes Stack
```

- **`CartContext`** — Gerencia o pedido ativo: produtos, pagamentos, cupom, split, email, gorjeta, mesa, comanda
- **`StoreContext`** — Carrega e mantém: whitelabel, configs, categorias, produtos, BOMs, dados de impressora
- **`TimerManager`** — Timer de inatividade (funcional mas menos crítico no PDV que no Totem)

### Fluxo de dados de uma venda

1. Operador monta pedido no `CartContext`
2. `Sale.mount()` monta o payload `ISale` com linhas, pagamentos, dados fiscais
3. Pedido é salvo no Firebase/Supabase para visibilidade no KDS
4. Pedido é cacheado localmente (proteção contra falhas)
5. `Sale.send()` envia ao endpoint de fila (`/sales`) do backend
6. Cron `Sale` (a cada 5s) processa a fila e confirma envio ao Odoo
7. DANFE/ticket são impressos conforme configuração

## 7. Regras de Negócio

- **Pagamentos offline são protegidos por senha** no modo PDV — o gerente deve autorizar pagamentos manuais
- **Edição de preço tem limites** configuráveis (`limitPrice`, `limitDiscount`, `restrictPriceControl`)
- **Pedidos com erro** são cacheados e reprocessados pelo cron `SaleError`
- **Divisão de conta** permite múltiplos pagamentos parciais até cobrir o valor total + gorjeta
- **Recuperação pós-crash** é automática — o app detecta pedidos pendentes em cache na reabertura
- **Validação de cupom** é executada automaticamente quando produtos do carrinho mudam
- **Pedidos rascunho** salvos no Firebase são visíveis no KDS para preparação antecipada

## 8. Deploy & Distribuição

### Build
- `yarn project playstore` → seleciona módulo Android genérico
- `yarn release:apk` ou `yarn release:bundle` para builds de produção
- Módulo `playstore` usa `flavorDimensions` para API 24/26 (SDKs Sumup diferentes)

### Distribuição
- **Google Play Store** — canal principal para dispositivos Android genéricos
- **APK direto** — distribuição via `yarn server:apk` para instalação manual

### Testes
- **Jest** para testes unitários
- **Maestro** para testes E2E (fluxos completos de totem, caixa e KDS)

## 9. Estado Atual & Observações

- O PDV é o modo mais maduro e feature-complete do NxzGo
- A integração com 6+ provedores de pagamento demonstra alta flexibilidade, mas também alta complexidade de manutenção
- O sistema de crons garante resiliência: pedidos nunca são perdidos, mesmo com falhas de rede
- A recuperação pós-crash (`isFirstOpenAfterRestart`) é um diferencial importante para operação em campo
- O sistema fiscal (SAT/NFC-e) adiciona complexidade significativa mas é requisito regulatório brasileiro
- A separação entre pagamentos "online" e "offline" com proteção por senha é uma regra de negócio crítica para evitar fraudes
