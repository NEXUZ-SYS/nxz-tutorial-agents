# Product Briefing: NxzGo — Modo SMART POS (Terminais Inteligentes)

> Gerado em 2026-03-16 por análise do codebase (v5.13.1).

## 1. Visão Geral

O **modo Smart POS** não é um "modo" de interface separado no sentido de uma tela diferente — é uma **variante de build e hardware** do NxzGo que roda em terminais de pagamento inteligentes (maquininhas com Android) da Cielo, Stone, PagBank e terminais MSiTef. A interface é a mesma do PDV ou Totem, mas com integrações nativas de pagamento e impressão específicas de cada fabricante.

O Smart POS transforma a maquininha de cartão em um terminal completo de PDV: o atendente monta o pedido, processa o pagamento diretamente na maquininha e imprime o comprovante — tudo em um único dispositivo, sem necessidade de tablet ou celular separado.

Cada fabricante tem seu próprio **módulo Android** (`android/cielo/`, `android/stone/`, `android/pagbank/`) com `applicationId`, SDKs nativos e configurações de assinatura distintas. A seleção é feita via `yarn project <modulo>` antes do build.

## 2. Público-Alvo

- **Atendentes de delivery** — Recebem pagamento na entrega com a maquininha
- **Operadores de balcão em operações compactas** — Estabelecimentos pequenos que usam a maquininha como PDV completo
- **Food trucks e ambulantes** — Solução all-in-one sem infraestrutura fixa
- **Garçons com maquininha** — Atendimento e pagamento na mesa

### Necessidades:
- Dispositivo único para pedido + pagamento + impressão
- Pagamento nativo no hardware (sem APIs externas ou Bluetooth)
- Impressora térmica integrada no dispositivo
- Interface compacta para telas menores de maquininhas

## 3. Funcionalidades Core

### 3.1 Detecção Automática de Flavor

O sistema detecta automaticamente o tipo de terminal via `NativeModules.ConfigModule.getConstants().flavor`:

| Flavor | Fabricante | `applicationId` | Versão |
|---|---|---|---|
| `cielo` | Cielo | `com.nxz.go` | 1.2.2 |
| `stone` | Stone | `br.com.nxz.go` | 1.0.0 |
| `pagbank` | PagBank | `com.moober_self_checkout` | 1.1.1 |
| `msitef` | Software Express (MSiTef) | — | — |

### 3.2 Pagamento Nativo

A grande diferença do Smart POS é o **pagamento nativo via hardware**, sem necessidade de APIs REST ou polling:

**`NativePaymentAdapter`** — Base para todos os provedores nativos:
- Usa `NativeModules.PaymentModule` + `NativeEventEmitter`
- Converte valor para centavos inteiros antes de enviar ao SDK nativo
- Recebe eventos de progresso via listener `PAYMENT`
- Suporta `USE_ACTIVE_CODE` para código de ativação do terminal

**Provedores nativos:**

| Provedor | Mecanismo | Métodos Suportados |
|---|---|---|
| **Cielo Native** | `PaymentModule.payment()` | Crédito, Débito, Voucher, PIX |
| **Stone Native** | `PaymentModule.payment()` | Crédito, Débito, Voucher, PIX |
| **PagBank Native** | `PaymentModule.payment()` + PlugPag SDK | Crédito, Débito, Voucher, PIX |
| **MSiTef** | `PaymentModule.payment()` + `configure()` | Crédito, Débito, Voucher + Admin Menu |

**Fluxo de pagamento nativo:**
1. App chama `paymentModule.payment(amount, transactionType, installments, activeCode?)`
2. SDK nativo assume o controle da tela para inserção/aproximação do cartão
3. Eventos `PAYMENT` são emitidos com progresso
4. Ao finalizar, retorna resultado ao app
5. App registra pagamento e prossegue com o pedido

### 3.3 MSiTef — Funcionalidades Específicas

O MSiTef tem funcionalidades extras não presentes em outros terminais:

- **`configure()`** — Inicialização do terminal MSiTef com parâmetros de conexão
- **`adminMenu()`** — Menu administrativo para operações de manutenção
- **Cancelamento de transação** — `cancelTransaction(IMSitefTransaction)` com modal de confirmação
- **Cache de transações** — Transações são salvas localmente e podem ser marcadas como canceladas via `markAsCancelled()`
- **Tela `MSitefCancellation`** — Disponível no drawer para cancelar transações anteriores

Erros do MSiTef (-1, -2, -6, -15) são tratados como cancelamento pelo usuário.

### 3.4 Impressão Nativa (V3)

Os Smart POS usam a **V3 do sistema de impressão**, que acessa a impressora térmica integrada do terminal:

| Terminal | Mecanismo | Detalhes |
|---|---|---|
| **Stone** | `NativeModules.PrinterModule.print()` | Envia JSON com operações, comunica via deep link Stone |
| **Cielo** | Native module proprietário | Encoder específico para hardware Cielo |
| **PagBank** | PlugPag SDK (`wrapper:1.30.51`) | Via SDK PagSeguro/PagBank |

A seleção é automática: se `NativeModules.ConfigModule.flavor` é `cielo`, `stone` ou `pagbank`, o factory da impressora instancia V3.

**Formatos de impressão:**
- **DANFE** — Nota fiscal completa com tabela de produtos, totais, pagamentos, troco, QR code do NF-e
- **Receipt** — Comprovante resumido
- **Ticket** — Ticket de produção para cozinha
- **Sticker** — Etiqueta de identificação
- **Test** — Página de teste da impressora

### 3.5 Interface Compacta

A detecção `useDeviceType()` identifica o Smart POS como dispositivo **phone** (tela pequena), ativando:
- Layout `PhoneView` do `ProductsList` — categorias em chips horizontais, grid compacto
- Carrinho otimizado para scroll vertical
- Teclado numérico adaptado

### 3.6 Build System Multi-módulo

Cada terminal tem seu próprio módulo Android completo:

```
android/
  ├── cielo/        # Cielo Smart POS
  │   ├── build.gradle (applicationId: com.nxz.go, v1.2.2)
  │   ├── src/main/java/  (módulos nativos Cielo)
  │   └── keystores/
  ├── stone/        # Stone Smart POS
  │   ├── build.gradle (applicationId: br.com.nxz.go, v1.0.0)
  │   ├── src/main/java/  (módulos nativos Stone)
  │   └── keystores/
  ├── pagbank/      # PagBank Smart POS
  │   ├── build.gradle (applicationId: com.moober_self_checkout, v1.1.1)
  │   ├── src/main/java/  (módulos nativos PagBank)
  │   └── libs/  (PlugPag SDK)
  ├── playstore/    # Build genérico
  └── sample/       # Desenvolvimento
```

Build para cada terminal:
```bash
yarn project cielo    # Cria symlink android/app → android/cielo
yarn release:apk      # Compila APK para Cielo Smart POS
```

## 4. Stack Técnica

### Hardware Suportado
- **Cielo LIO** — Terminal Android da Cielo
- **Stone POS** — Terminal Android da Stone
- **PagBank Moderninha Smart** — Terminal Android PagBank (PlugPag SDK)
- **Terminais MSiTef** — Qualquer terminal compatível com Software Express MSiTef

### Frameworks Nativos
- **React Native 0.81.1** com módulos nativos por fabricante
- **NativeModules**: `PaymentModule`, `PrinterModule`, `ConfigModule`
- **NativeEventEmitter**: Comunicação assíncrona com SDK nativo

### SDKs por Fabricante
| Fabricante | SDK | Versão |
|---|---|---|
| PagBank | PlugPag Service Wrapper | 1.30.51 |
| Sumup (Playstore) | Sumup Android SDK | 4.2.0 / 5.0.4 |
| Stone | SDK proprietário (AAR) | — |
| Cielo | SDK proprietário | — |

### Infraestrutura de Build
- Cada módulo usa Hermes como JS engine
- Sentry integrado em todos os módulos (`sentry.gradle`)
- ProGuard/R8 para minificação em release
- Keystores separados por módulo para assinatura

## 5. Arquitetura

### Decisão Runtime do Provedor

```
App inicializa
  └── Payment constructor
       ├── Verifica NativeModules.PaymentModule
       │   ├── Se existe → carrega adapter Nativo (Cielo/Stone/PagBank/MSiTef)
       │   └── Se não existe → carrega adapters Playstore (APIs REST)
       └── Sempre carrega: Common (Cash)
```

```
Printer factory
  └── Verifica NativeModules.ConfigModule.flavor
       ├── 'cielo'/'stone'/'pagbank' → V3 (Native SmartPOS printer)
       ├── mode === 'direct' → V2 (BLE/Network/USB)
       └── default → V1 (RawBT/EscPOS)
```

### Diagrama de Comunicação Nativa

```
React Native JS
  ↕ NativeEventEmitter / NativeModules bridge
Java/Kotlin Native Module
  ↕ SDK-specific API
Hardware (chip reader / thermal printer / NFC)
```

## 6. Integrações

| Integração | Finalidade | Detalhes Smart POS |
|---|---|---|
| **SDK Nativo de Pagamento** | Pagamento integrado no hardware | Sem latência de rede, sem polling |
| **Impressora Térmica Nativa** | Impressão direta no papel do terminal | Sem configuração de impressora externa |
| **Odoo ERP** | Backend (sessão, catálogo, vendas) | Mesmo de todos os modos |
| **Firebase/Supabase** | Real-time para KDS | Mesmo de todos os modos |
| **SAT/NFC-e** | Fiscal | Mesmo de todos os modos |
| **Sentry** | Monitoramento | Essencial — terminais operam em campo |

## 7. Regras de Negócio

- **Pagamento nativo é instantâneo** — Sem necessidade de polling HTTP; o SDK nativo controla todo o fluxo de pagamento
- **Impressão é mandatória** — Smart POS sempre tem impressora integrada; o comprovante é impresso automaticamente
- **Cancelamento MSiTef tem cache local** — Transações são rastreadas para permitir cancelamento posterior
- **Build por dispositivo é obrigatório** — Cada terminal requer seu APK específico (`yarn project <módulo>`)
- **`USE_ACTIVE_CODE`** — Alguns terminais requerem código de ativação para processar pagamentos
- **Valores em centavos** — O `NativePaymentAdapter` converte valores para inteiros antes de enviar ao SDK nativo
- **Modo `BOTH` funciona em Smart POS** — O terminal pode alternar entre PDV e KDS
- **Release signing para debug em Cielo** — O módulo Cielo usa configuração de release para builds de debug (requisito do SDK Cielo)

## 8. Deploy & Distribuição

### Build Process
```bash
# 1. Selecionar módulo do terminal
yarn project cielo    # ou stone, pagbank

# 2. Build
yarn release:apk      # APK para instalação direta

# 3. Distribuição
# APKs são instalados diretamente nos terminais
# Não passam pela Play Store
```

### Distribuição
- **APK direto** — Instalação via USB, MDM (Mobile Device Management) ou OTA
- **Não via Play Store** — Terminais Smart POS têm lojas proprietárias ou instalação gerenciada
- **OTA updates** — Cron `Upgrade` verifica atualizações disponíveis
- `yarn server:apk` — Serve APKs via HTTP para download local

### Versionamento
Cada módulo tem versionamento independente:
- `cielo`: 1.2.2
- `stone`: 1.0.0
- `pagbank`: 1.1.1
- `playstore`: 5.19.9

### Testes
- **Jest** para testes unitários (lógica compartilhada)
- **Maestro** para E2E (funciona em emuladores, não em Smart POS real)
- Testes em dispositivo real são necessários para validar integrações nativas

## 9. Estado Atual & Observações

### Pontos fortes
- Arquitetura de adapters bem desenhada: `NativePaymentAdapter` e `NativeEventEmitter` abstraem diferenças entre fabricantes
- Factory pattern no printer (V1/V2/V3) e no payment (Common/Playstore/Native) permite extensibilidade
- O mesmo app React Native roda em todos os terminais com código compartilhado
- Cache de transações MSiTef permite cancelamento posterior (requisito operacional)
- Build system multi-módulo com symlinks é elegante e eficiente

### Áreas de atenção
- **Versionamento divergente** — `playstore` está em 5.19.9 enquanto `stone` está em 1.0.0; isso pode causar confusão sobre qual versão está em qual terminal
- **Testes E2E em Smart POS** — Maestro não roda nativamente em terminais; testes manuais são necessários
- **SDKs proprietários** — Atualizações de SDK dos fabricantes podem quebrar integrações; cada atualização requer validação por terminal
- **Tela pequena** — Smart POS tipicamente tem tela de ~5.5", a UX precisa ser validada continuamente para usabilidade
- **Performance em hardware limitado** — Maquininhas têm processadores e memória inferiores a tablets/phones; React Native + Hermes ajuda, mas é um constraint importante
- **Certificação e homologação** — Cada fabricante tem seu processo de homologação para publicar na loja do terminal; `docs/homologacao/` sugere que isso está em andamento
- **Módulo MSiTef** é o mais complexo — `configure()`, `adminMenu()`, cache de transações, e cancellation flow adicionam camadas de complexidade
